────────────────────────────────────────────────────────────────────────────────
  OVERVIEW
────────────────────────────────────────────────────────────────────────────────

VoiceForge is a local voice-controlled AI agent for developers. Users speak a
natural language command (e.g. "Create a Python file called hello.py that prints
Hello World"), and the system transcribes it, understands the intent, builds a
structured execution plan, asks the user to review/edit it, and then safely
executes tool actions — all within a sandboxed output/ directory.

────────────────────────────────────────────────────────────────────────────────
  STEP-BY-STEP EXECUTION FLOW
────────────────────────────────────────────────────────────────────────────────

STEP 1 — AUDIO INPUT
─────────────────────
  Source:   app.py (Streamlit UI)
  What:     User speaks a command via the browser microphone OR uploads a
            pre-recorded WAV/MP3/M4A file.
  How:      `audio_recorder_streamlit` captures raw audio bytes from the browser.
            An uploaded file is read directly as bytes.
  Result:   Raw audio bytes ready for processing.

──────────────────────────────────────────────────────────────────────────
STEP 2 — WRITE AUDIO TO TEMP FILE
───────────────────────────────────
  Source:   app.py → run_audio_pipeline()
  What:     Audio bytes are written to a temporary file on disk.
  How:      Python's `tempfile.NamedTemporaryFile` creates a disposable .wav
            or .mp3 file. Its path is passed to the STT engine.
  Result:   Temp file at a known path e.g. C:/Users/.../tmpXXXX.wav

──────────────────────────────────────────────────────────────────────────
STEP 3 — SPEECH-TO-TEXT (STT)
───────────────────────────────
  Source:   utils/stt.py → transcribe_audio()
  What:     The temporary audio file is transcribed to text.
  How (Fast path):
            If USE_LOCAL_STT=false (default), the audio file is sent to
            Groq's API endpoint using the OpenAI Python SDK:
              - Model: whisper-large-v3-turbo
              - Returns: plain-text transcription in ~1-2 seconds
  How (Local path):
            If USE_LOCAL_STT=true, the `faster-whisper` library loads the
            "base" Whisper model locally on CPU (requires C++ Build Tools).
              - Much slower (~2-10 min on CPU)
              - More private (no data leaves the machine)
  Cleanup:  Temp file is deleted after transcription.
  Result:   String transcript e.g. "Create a python file called hello.py
            that prints hello world"

──────────────────────────────────────────────────────────────────────────
STEP 4 — INTENT CLASSIFICATION & PLAN GENERATION (LLM)
─────────────────────────────────────────────────────────
  Source:   utils/llm.py → detect_intent_and_extract()
  What:     The transcript + last 4 turns of session memory are sent to an LLM.
            The LLM classifies the intent and generates a structured JSON plan.
  How (Fast/Default):
            If USE_API_LLM=true, uses Groq API:
              - Model: llama-3.1-8b-instant
              - Returns JSON in ~1 second
  How (Local):
            If USE_API_LLM=false, uses local Ollama:
              - Model: configurable via LOCAL_LLM_MODEL in .env (e.g. qwen3.5)
              - Slower; requires Ollama running (`ollama serve`)
  Memory:   Last 4 user+agent messages injected for compound/follow-up commands.
  Output schema:
    {
      "intent": "compound_command" | "general_chat" | "summarize" | "unknown",
      "steps": [
        {
          "action": "write_code" | "create_file" | "append_code" | "none",
          "path": "filename.ext",
          "content": "...full file content..."
        }
      ],
      "requires_confirmation": true | false,
      "response": "Brief agent message"
    }
  Security: System prompt contains an injection guard that explicitly
            instructs the LLM to treat the transcript as DATA only, not
            as new instructions. Prevents prompt hijacking.
  Result:   Structured plan dict

──────────────────────────────────────────────────────────────────────────
STEP 5 — SAFETY VALIDATOR (AUTO)
──────────────────────────────────
  Source:   app.py (pipeline logic) + utils/tools.py
  What:     All file paths extracted from the LLM plan are automatically
            sanitized. This happens silently before any execution.
  How:      os.path.basename() strips directory traversal attacks.
            os.path.join(OUTPUT_DIR, safe_name) locks every write to /output/.
            Input like "../../system/secrets.json" becomes "secrets.json"
            inside the safe output/ directory.
  Result:   Paths validated. Execution trace shows "🛡️ Safety Validator ✅ passed"

──────────────────────────────────────────────────────────────────────────
STEP 6 — HUMAN-IN-THE-LOOP REVIEW (if file operations detected)
─────────────────────────────────────────────────────────────────
  Source:   app.py → "awaiting" pipeline stage
  What:     When the plan contains file-writing steps, the UI pauses and
            presents the full plan for the user to review and edit before
            any execution occurs.
  How:      Each step is rendered as an expandable card in the UI with:
              - Dropdown: change the action type
              - Text input: rename the file
              - Text area: edit or rewrite the generated code/content
  Options:
              ✅ Approve & Execute  — runs all edited steps
              ❌ Cancel Plan        — aborts, nothing written to disk
  Result:   User has full editorial control over what gets written locally.

──────────────────────────────────────────────────────────────────────────
STEP 7 — TOOL EXECUTION
─────────────────────────
  Source:   utils/tools.py → execute_plan_step()
  What:     Each approved step in the plan is executed as a real local action.
  Actions:
    write_code   → opens/creates file, writes full content
    create_file  → creates file (empty or with specified content)
    append_code  → opens existing file in append mode, adds new content
  All writes:   os.path.join(OUTPUT_DIR, os.path.basename(path))
                NOTHING is written outside D:\Voice-agent\output\
  Result:   Physical files created on disk. Status returned per step.

──────────────────────────────────────────────────────────────────────────
STEP 8 — ARTIFACT RENDERING & DOWNLOAD
────────────────────────────────────────
  Source:   app.py → "done" pipeline stage
  What:     All files generated in the session are rendered in tabbed preview.
  How:      st.code() renders syntax-highlighted code.
            st.download_button() provides instant download.
            /output file tree is listed with file sizes.
  Memory:   All created file contents remain in session_state.artifacts
            for the duration of the session.
  Result:   User can inspect, copy, re-download any generated file.

──────────────────────────────────────────────────────────────────────────
STEP 9 — SESSION MEMORY PERSISTENCE
─────────────────────────────────────
  Source:   Sidebar (app.py) + chat_history in session_state
  What:     Every user command and agent response is logged into a rolling
            session memory visible in the sidebar.
  How:      chat_history list is passed into the LLM on every new request.
            LLM uses last 4 turns to understand follow-up commands like:
              "Now append a retry function to that file"
              "Summarize the same text more briefly"
  Result:   The agent feels stateful and aware of prior context.

────────────────────────────────────────────────────────────────────────────────
  SUPPORTED INTENT TYPES
────────────────────────────────────────────────────────────────────────────────

  compound_command  → Multiple steps (create + write, etc.) in one command
  general_chat      → Conversational question (no file ops)
  summarize         → Summarizes provided text (no file ops)
  unknown           → Falls back gracefully, asks for clarification

────────────────────────────────────────────────────────────────────────────────
  SAFETY ARCHITECTURE SUMMARY
────────────────────────────────────────────────────────────────────────────────

  1. Path Sandboxing      → output/ only, os.path.basename enforced
  2. Prompt Injection     → SECURITY DIRECTIVE in system prompt
  3. Human Approval       → No disk write without explicit user approval
  4. API Key Safety       → .env file, .gitignore prevents GitHub exposure
  5. Graceful Degradation → All errors caught, displayed in UI, never crash
